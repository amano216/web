Gitの歴史

Linuxカーネル開発の危機から生まれ、世界のソフトウェア開発を変革した分散型バージョン管理システムGitの誕生と進化の物語。

Git誕生前夜 - バージョン管理の歴史

1972年

SCCS（Source Code Control System）

Bell研究所で開発された最初期のバージョン管理システム。ファイルの変更履歴を保存できたが、1つのファイルを1人しか編集できないロック方式。

1982年

RCS（Revision Control System）

SCCSの改良版。より効率的な差分保存方式を採用。しかし依然として単一ファイル管理で、チーム開発には不向き。

1990年

CVS（Concurrent Versions System）

複数人での同時編集を可能にした革新的システム。ネットワーク経由でのアクセスも実現。しかし、ファイル移動や名前変更の追跡ができないなど、多くの制限があった。

2000年

Subversion（SVN）

「CVSの正統な後継者」を目指して開発。ディレクトリのバージョン管理、アトミックなコミット、効率的なブランチングなどを実現。しかし中央集権型の限界は残る。

Gitの誕生 - Linuxカーネル開発の危機

2002-2005年

BitKeeperの採用と決裂

Linuxカーネル開発では、商用の分散型VCS「BitKeeper」を使用していた。しかし2005年、リバースエンジニアリング問題により、BitKeeper社が無償利用を打ち切り。Linus Torvaldsは新しいVCSが必要に。

Linusが求めた要件：
  • 分散型であること（SVNのような中央サーバー不要）
  • 高速であること（特にブランチ作成）
  • シンプルな設計
  • 大規模プロジェクトに対応
  • データの完全性を保証
2005年4月3日

Gitの開発開始

Linus Torvaldsが既存のVCSに失望し、自らGitの開発を開始。「愚かなコンテンツトラッカー」（stupid content tracker）と自虐的に命名。わずか4日後には自己ホスティングを開始。

# 最初のGitコミット（2005年4月7日） commit e83c5163316f89bfbde7d9ab23ca2e25604af290 Author: Linus Torvalds <torvalds@ppc970.osdl.org> Date: Thu Apr 7 15:13:13 2005 -0700 Initial revision of "git", the information manager from hell
2005年6月

Gitでのカーネル開発開始

わずか2ヶ月でLinuxカーネル開発に使用できるレベルに到達。7月にはJunio C Hamano（濱野純）がメンテナーを引き継ぎ、Linusはカーネル開発に専念。

Gitの設計哲学

2005年

革新的な設計思想

Gitは従来のVCSとは根本的に異なる設計を採用。ファイルの「差分」ではなく「スナップショット」を保存し、すべてのオブジェクトをSHA-1ハッシュで管理。

従来のVCS Git
差分（デルタ）を保存 スナップショットを保存
中央サーバー必須 完全分散型
ブランチはディレクトリのコピー ブランチは単なるポインタ
履歴の改変が可能 暗号学的に履歴を保護
2005-2008年

基本機能の確立

rebase、cherry-pick、bisectなど、現在も使われる重要な機能が次々と実装される。特にrebaseは、きれいな履歴を保つための革新的機能として注目される。

# 初期のGitコマンド例 git init git add . git commit -m "Initial commit" git branch experimental git checkout experimental git merge master # 革新的だったrebase git rebase master

GitHubとGitエコシステムの爆発

2008年4月

GitHub登場

Tom Preston-Werner、Chris Wanstrath、PJ HyettがGitHubを立ち上げ。「Social Coding」のコンセプトで、プルリクエスト、Issues、Wikiなどの機能を統合。Gitの普及に決定的な役割を果たす。

GitHubがもたらした革新：
  • プルリクエスト - コードレビューの文化を確立
  • フォーク - オープンソース貢献の敷居を下げる
  • ソーシャル機能 - 開発者のポートフォリオとして機能
  • GitHub Pages - 静的サイトホスティング
  • 統合サービス - CI/CDとの連携
2010年代前半

Git採用の加速

Google（2011年）、Microsoft（2017年）など、大手企業が次々とGitを採用。分散型の利点、高速性、柔軟なワークフローが評価される。

2018年6月

MicrosoftがGitHubを買収

75億ドルでMicrosoftがGitHubを買収。オープンソースに対する姿勢の変化を象徴する出来事。GitHubは独立性を保ちながら、さらなる成長を遂げる。

Gitの進化と現代

2010年代後半

パフォーマンスの改善

大規模リポジトリ対応のため、様々な最適化が実施される。Microsoftは「VFS for Git」を開発し、300GBを超えるWindows開発でもGitを使用可能に。

改善項目 内容
部分クローン 必要なオブジェクトのみダウンロード
プロトコルv2 より効率的な通信
コミットグラフ 履歴探索の高速化
マルチパックインデックス 大規模リポジトリの最適化
2020年代

SHA-256への移行準備

SHA-1の脆弱性に対応するため、SHA-256への移行が進行中。後方互換性を保ちながら、より安全なハッシュアルゴリズムへの移行を計画。

現在

Gitの普遍化

Stack Overflow調査では90%以上の開発者がGitを使用。GitLab、Bitbucket、Giteaなど多様なプラットフォームが登場し、エコシステムはさらに拡大。

現代のGitの特徴：
  • デファクトスタンダード - バージョン管理の事実上の標準
  • 多様なワークフロー - Git Flow、GitHub Flow、GitLab Flowなど
  • 統合開発環境 - IDE、エディタに組み込まれたGit機能
  • 自動化 - CI/CD、bot、自動マージなど
  • 教育 - 大学のCS教育でも標準的に教えられる

Gitの歴史から学ぶこと

Gitの成功が教える教訓：
  1. 必要は発明の母 - 実際の問題を解決するために生まれたツールは強い
  2. シンプルで強力な設計 - 複雑な操作も基本的な概念の組み合わせで実現
  3. 分散型の思想 - 中央集権から分散へ、これは技術トレンドの先取り
  4. エコシステムの重要性 - GitHubなどのプラットフォームが普及を加速
  5. オープンソースの力 - コミュニティによる継続的な改善

バージョン管理システムの進化

世代 代表例 特徴 限界
第1世代 SCCS, RCS ローカル、単一ファイル チーム開発困難
第2世代 CVS, SVN 中央サーバー、同時編集 オフライン作業不可
第3世代 Git, Mercurial 分散型、高速ブランチ 学習曲線が急

「2週間で作る」と言って始まったGitは、今や世界中の開発者にとって欠かせないツールとなりました。Linuxカーネルという巨大プロジェクトの実践的なニーズから生まれ、シンプルながら強力な設計により、あらゆる規模のプロジェクトに対応できるシステムへと成長しました。Gitの歴史は、優れた設計とコミュニティの力が組み合わさったとき、いかに革新的な変化を起こせるかを示す最良の例です。
