プログラミング命名規則チートシート

プログラミングにおける変数、関数、クラスなどの命名規則をまとめたリファレンスです。可読性が高く、保守しやすいコードを書くための基本的なガイドラインを提供します。

命名のベストプラクティス

一般的な原則

原則 説明 良い例 / 悪い例
意味のある名前 変数や関数の目的が明確に分かる名前を使用 ✅ userAge
❌ a, temp
検索しやすい名前 コード内で検索しやすい、一意性のある名前 ✅ MAX_RETRY_COUNT
❌ MAX
発音しやすい名前 チーム内で口頭で議論しやすい名前 ✅ customerRecord
❌ xCstmrRcrd
一貫性 プロジェクト全体で同じ命名規則を使用 ✅ 全て getUserById
❌ 混在: fetchUser, retrieveUser
適切な長さ 短すぎず長すぎない、バランスの取れた名前 ✅ calculateTotalPrice
❌ calc, calculateTheTotalPriceOfAllItemsInTheShoppingCart

避けるべきパターン

❌ 悪い例:

// 意味不明な略語
const d = new Date();
const yrs = calcYrs();

// 数字を含む変数名
const value1 = "John";
const value2 = "Jane";

// 誤解を招く名前
const accountList = {}; // 実際はオブジェクト

// ノイズワード
const theUser = {};
const userData = {};
const userInfo = {};

✅ 良い例:

// 明確で意味のある名前
const currentDate = new Date();
const yearsSinceJoined = calculateYearsSinceJoined();

// 具体的な名前
const firstName = "John";
const lastName = "Jane";

// 正確な型を示す名前
const accountMap = {};

// 明確に区別された名前
const user = {};
const userProfile = {};
const userSettings = {};

特殊なケース

ブール値の命名

パターン 例 説明
is + 形容詞/名詞 isActive, isValid, isUser 状態を表す最も一般的なパターン
has + 名詞 hasPermission, hasChildren 所有や包含を表す
can + 動詞 canEdit, canDelete 能力や権限を表す
should + 動詞 shouldUpdate, shouldRender 推奨される動作を表す

コレクションの命名

型 単数形 複数形/コレクション
配列 user users, userList, userArray
マップ/辞書 user userMap, userById, userDictionary
セット tag tagSet, uniqueTags

種類 パターン 例
