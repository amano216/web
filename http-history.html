HTTPの歴史

インターネットの基盤プロトコルであるHTTPが、シンプルな文書転送から現代のWebアプリケーションを支える高度なプロトコルへと進化した軌跡を辿ります。

HTTPの誕生

1989-1991年

HTTP/0.9 - ワンラインプロトコル

Tim Berners-LeeがWorld Wide Webの一部として設計した最初のHTTP。極めてシンプルで、GETメソッドのみ、ヘッダーなし、HTMLのみ返すという仕様。リクエストは文字通り1行だけだった。

GET /index.html （レスポンスはHTMLのみ、ステータスコードもヘッダーもなし） <html> A very simple HTML page </html>

接続は1リクエストごとに切断される設計で、現在から見ると信じられないほどシンプル。しかし、この単純さが初期のWeb普及に貢献した。

HTTPの標準化

1996年5月

HTTP/1.0 - 拡張性の追加

RFC 1945として正式に文書化。ヘッダー、ステータスコード、POST・HEADメソッド、Content-Typeによる多様なコンテンツのサポートなど、現在のHTTPの基礎となる機能が追加される。

GET /index.html HTTP/1.0 Host: www.example.com User-Agent: Mozilla/1.0 HTTP/1.0 200 OK Content-Type: text/html Content-Length: 1234 <html>...</html>
HTTP/1.0の新機能 意義
HTTPヘッダー メタデータの送信が可能に
ステータスコード エラー処理の標準化
Content-Type HTML以外のコンテンツも扱える
POST/HEADメソッド データ送信と効率的な確認
1997年1月

HTTP/1.1 - 現代HTTPの基盤

RFC 2068（後にRFC 2616）として標準化。持続的接続、チャンク転送、ホストヘッダー必須化、キャッシュ制御の改善など、インターネットの爆発的成長に対応する機能を搭載。

HTTP/1.1の革新的機能：
  • 持続的接続（Keep-Alive） - 1つのTCP接続で複数のリクエスト/レスポンス
  • パイプライニング - レスポンスを待たずに次のリクエストを送信
  • チャンク転送エンコーディング - コンテンツ長が不明でも送信開始可能
  • Hostヘッダー必須 - 1つのIPアドレスで複数のドメインをホスト
  • より洗練されたキャッシュ制御 - Cache-Control、ETag

Web 2.0時代の課題

2000年代

HTTP/1.1の限界

Ajax、リッチなWebアプリケーション、ソーシャルメディアの時代。1つのページで数十〜数百のリソースを読み込むようになり、HTTP/1.1の設計限界が露呈。

顕在化した問題：
  • HOLブロッキング - 1つの遅いリソースが全体を遅延させる
  • 同時接続数制限 - ブラウザは同一ドメインに6接続程度まで
  • 冗長なヘッダー - 同じヘッダーを何度も送信
  • ドメインシャーディング - 接続数制限を回避する苦肉の策
2009年

SPDYの実験

GoogleがHTTP/1.1の問題を解決するため、独自プロトコル「SPDY」を開発。多重化、ヘッダー圧縮、サーバープッシュなどの実験的機能を実装。後のHTTP/2の基礎となる。

HTTP/2の革命

2015年5月

HTTP/2 - バイナリプロトコルへ

RFC 7540として標準化。SPDYをベースに、パフォーマンスの大幅改善を実現。後方互換性を保ちながら、プロトコルレベルで革新的な変更を導入。

// HTTP/2はバイナリプロトコルのため、実際の通信は以下のような概念 [HEADERS frame] :method = GET :path = /index.html :scheme = https :authority = www.example.com user-agent = Mozilla/5.0 [DATA frame] （バイナリデータ） // 同時に複数のストリームが多重化される
HTTP/2の特徴 解決した問題
バイナリフレーミング パース効率の向上、エラー削減
ストリーム多重化 HOLブロッキングの解消
ヘッダー圧縮（HPACK） 帯域幅の節約
サーバープッシュ レイテンシの削減
ストリーム優先順位 重要なリソースを優先
2015-2020年

HTTP/2の普及

主要ブラウザとサーバーが迅速に対応。HTTPS必須という実質的な要件により、同時にWebの暗号化も促進。2020年には主要サイトの半数以上がHTTP/2を採用。

HTTP/3とQUICの時代

2013-2018年

QUICの開発

GoogleがTCPの限界を克服するため、UDP上に新しいトランスポートプロトコル「QUIC」を開発。0-RTTハンドシェイク、コネクションマイグレーション、改善された輻輳制御などを実現。

QUICが解決する問題：
  • TCPのHOLブロッキング - パケットロスが全ストリームに影響
  • ハンドシェイクの遅延 - TCP + TLSで複数往復が必要
  • 接続の継続性 - IPアドレスが変わると接続断
  • ミドルボックスの干渉 - TCPの拡張が困難
2022年6月

HTTP/3の標準化

RFC 9114として正式に標準化。QUIC上で動作し、さらなるパフォーマンス向上と信頼性を実現。モバイルネットワークなど、不安定な環境でも高速な通信を維持。

HTTP/3の利点 実現する機能
0-RTTハンドシェイク 再接続時の即座の通信開始
独立したストリーム 1つのパケットロスが他に影響しない
コネクションマイグレーション ネットワーク切り替え時も接続維持
改善された輻輳制御 ネットワーク状況により適応的

HTTPの未来

2020年代〜

次世代Webの要求

WebAssembly、WebRTC、IoT、エッジコンピューティングなど、新しい使用例がHTTPに新たな要求を突きつける。プロトコルは常に進化を続ける。

将来の課題と方向性：
  • 超低遅延通信 - リアルタイムアプリケーションの要求
  • 大規模IoT対応 - 軽量で効率的なプロトコル
  • プライバシー保護 - ECH（Encrypted Client Hello）など
  • 新しいトランスポート - 衛星通信、5G/6Gへの最適化
  • セマンティックWeb - より意味的な通信へ

HTTPの歴史から学ぶこと

HTTPの進化が教える教訓：
  1. シンプルさの力 - HTTP/0.9の単純さが初期のWeb普及を促進
  2. 後方互換性の重要性 - 新バージョンも既存のWebを壊さない設計
  3. 実装先行の価値 - SPDY、QUICなど、実験的実装が標準化を牽引
  4. レイヤー違反の勇気 - HTTP/3はトランスポート層まで再設計
  5. 継続的な改善 - 完璧を求めず、段階的に問題を解決

HTTPバージョンの比較

バージョン リリース年 主な特徴 解決した問題
HTTP/0.9 1991年 GETのみ、HTMLのみ 文書の共有を実現
HTTP/1.0 1996年 ヘッダー、メソッド追加 多様なコンテンツ対応
HTTP/1.1 1997年 持続的接続、Host必須 効率性、仮想ホスト
HTTP/2 2015年 バイナリ、多重化 HOLブロッキング、効率
HTTP/3 2022年 QUIC、0-RTT TCPの限界、モバイル対応

HTTPは単純な文書転送プロトコルから始まり、現代のインターネットを支える高度なプロトコルへと進化しました。その成功の鍵は、時代のニーズに応じて大胆に進化しながらも、既存のWebとの互換性を保ち続けたことにあります。HTTPの歴史は、技術の進化が段階的かつ実用的であることの重要性を示す、素晴らしい例です。
